# 智能仓储机器人调度系统 — 评测方案

> 分析范围：NL 解析、路径规划、冲突解决、重规划、部分执行、性能指标
> 不修改代码，仅设计方案

---

## 一、为什么需要评测体系

当前项目缺少系统化的评测能力，导致：

| 问题 | 表现 |
|------|------|
| **无法量化改进效果** | 改了一行代码，规划成功率是提升了还是下降了？不知道 |
| **无法发现回归** | 引入新功能后，旧场景是否仍然正确？不可知 |
| **LLM 输出不稳定** | 同一个指令，DeepSeek 今天和明天的解析结果可能不同——没有评测就抓不住这种退化 |
| **无法对比方案** | A* 参数调优、重规划策略调整、prompt 修改——哪个更好？需要定量对比 |

## 二、评测指标

### 2.1 指标总览

```
评测指标
├── 规划成功率指标
│   ├── 路径规划成功率 (全局)
│   ├── 初次规划成功率 (0次重规划)
│   ├── 重规划后成功率 (1-3次)
│   └── 部分执行触发率
│
├── 路径质量指标
│   ├── 平均路径长度 / 平均 makespan
│   ├── 平均等待步数
│   ├── 平均计算节点展开数
│   └── 最优性差距 (vs 理论最短)
│
├── NL理解指标
│   ├── 意图分类准确率
│   ├── 任务解析准确率 (精确匹配)
│   ├── 任务解析容错率 (部分匹配)
│   └── 实体提取 F1 (机器人ID、位置、约束)
│
├── 性能指标
│   ├── 平均总规划耗时
│   ├── 平均 LLM 调用耗时
│   ├── 平均 A* 调用耗时
│   ├── P50 / P95 / P99 延迟
│   └── Token 消耗 (输入 + 输出)
│
└── 鲁棒性指标
    ├── 冲突场景解决率
    ├── 死锁场景恢复率
    ├── 无效输入拒绝率
    └── LLM 降级触发率
```

### 2.2 核心指标详解

#### 指标 1：路径规划成功率

```
公式: 规划成功任务数 / 用户输入任务总数
```

这是项目 AGENTS.md 已确认的 MVP 指标。需要区分三个粒度：

| 粒度 | 说明 | 计算方式 |
|------|------|---------|
| **批次级** | 整个调度请求是否成功 | 全部任务有路径 = success |
| **任务级** | 每个机器人是否成功 | 单机器人有路径 = success |
| **场景级** | 固定场景下的统计 | 多批次测试的平均值 |

#### 指标 2：初次规划成功率

```
公式: 0 次重规划即成功的任务数 / 总任务数
```

反映 A* 规划质量和初始优先级排序的好坏。初次成功率越高，说明：
- 优先级排序策略越好（LLM 理解用户意图更准确）
- 地图和机器人初始位置越合理
- A* 启发函数越有效

#### 指标 3：重规划效率

```
指标:
- 平均重规划次数 (0 ~ 3)
- 重规划后冲突解决率
- 重规划增加路径长度比例
```

衡量冲突诊断和重规划策略的有效性。

#### 指标 4：NL 理解准确率

需要区分三个子任务：

**4a. 意图分类准确率**

```
测试: 给 100 条标注好意图的指令，对比 LLM 分类结果
指标: Accuracy = 正确分类数 / 总指令数

意图类别: schedule / cron_create / cron_list / cron_delete /
          cron_toggle / general_qa / show_map / robot_move /
          map_modify / cargo_done
```

**4b. 任务解析准确率**

```
测试: 给 100 条标注好结构化任务的调度指令
指标:
- 精确匹配率: robot_id + goal_location_id + priority 全部正确的比例
- 部分匹配率: robot_id + goal_location_id 正确的比例
- 实体 F1: 机器人ID、目标位置、约束条件的提取 F1
```

**4c. 位置解析准确率**

```
测试: 位置别名 → 坐标的解析是否正确
指标: 位置匹配准确率
```

#### 指标 5：冲突检测与解决

```
指标:
- 冲突检测召回率: 实际存在的冲突中检测出的比例
- 冲突检测精确率: 报告的冲突中确实存在的比例
- 冲突解决率: 检测到的冲突中成功解决的比例
- 误解决率: 重规划后引入新冲突的比例
```

#### 指标 6：部分执行质量

```
指标:
- 部分执行触发率: 多少批次触发了部分执行
- 最大化完成度: 部分执行后完成任务数 / 总任务数
- 高优先级保留率: 高优任务是否优先保留
```

#### 指标 7：性能指标

```
指标:
- 端到端延迟: 用户输入 → 返回结果 (区分 LLM 调用 + A* 规划)
- LLM 调用延迟: classify_intent + parse (旧路径) vs 一次调用 (新路径)
- A* 规划延迟: 总展开节点数 / 调用次数
- Token 消耗: 每批次输入/输出 tokens
```

#### 指标 8：鲁棒性指标

```
指标:
- 无效输入容错: 乱码 / 空输入 / 不相关输入 → 正确拒绝或降级
- LLM 调用失败恢复: API 超时 / 格式错误 → 正确降级
- 极端场景: 所有机器人同一起点 / 同一目标 / 通道全封闭
```

需要构建三类数据集：

### 3.2 评测执行方法

#### 方法 1：离线评测 (batch_eval.py)

```
输入: 评测数据集 (JSON)
          │
          ▼
    ┌─────────────────────┐
    │  评测调度器           │
    │  for each case:      │
    │    workflow.run()     │
    │    collect_metrics() │
    └──────────┬──────────┘
               │
               ▼
    ┌─────────────────────┐
    │  结果对比器           │
    │  actual vs expected  │
    │  compute_metrics()   │
    └──────────┬──────────┘
               │
               ▼
    ┌─────────────────────┐
    │  报告生成器           │
    │  JSON + Markdown     │
    │  历史趋势对比        │
    └─────────────────────┘
```

**执行流程**：

```python
def run_evaluation(dataset_path: str) -> dict:
    """离线评测：加载数据集 → 逐条执行 → 收集指标 → 生成报告。"""
    dataset = load_dataset(dataset_path)
    results = []

    for case in dataset:
        # 1. 执行调度
        state = workflow.run(case["instruction"])

        # 2. 收集指标
        metrics = {
            "case_id": case["id"],
            "category": case.get("category", "unknown"),
            "intent": extract_intent(state),
            "success_rate": state.metrics.planning_success_rate,
            "replan_count": state.retry_count,
            "total_time_ms": state.total_planning_time_ms,
            "conflict_count": len(state.current_conflicts),
            "llm_calls": count_llm_calls(),
            "token_usage": count_tokens(),
        }

        # 3. 对比预期结果
        comparison = compare_with_expected(metrics, case.get("expected", {}))
        results.append({"case": case, "metrics": metrics, "comparison": comparison})

    # 4. 生成汇总报告
    report = generate_report(results)
    return report
```

#### 方法 2：A/B 对比评测

```python
def ab_compare(branch_a: str, branch_b: str, dataset_path: str) -> dict:
    """对比两个版本的评测结果。"""
    # 运行 A 版本
    results_a = checkout_and_run(branch_a, dataset_path)
    # 运行 B 版本
    results_b = checkout_and_run(branch_b, dataset_path)
    # 对比
    diff = compare_results(results_a, results_b)
    return diff
```

对比维度：
- 同一数据集、不同 prompt 版本
- 同一数据集、不同重规划策略
- 同一数据集、不同 LLM 模型（DeepSeek vs GPT-4o）

### 3.3 评分与通过标准

#### 等级评分

```python
def score_case(metrics: dict, expected: dict) -> str:
    """单条用例评分: PASS / WARN / FAIL"""
    if metrics["status"] == "error":
        return "FAIL"
    if metrics["planning_success_rate"] < expected.get("min_success_rate", 0.8):
        return "FAIL"
    if metrics["replan_count"] > expected.get("max_replans", 3):
        return "WARN"
    return "PASS"
```

#### 通过标准建议

| 指标 | MVP 目标 | 正式版目标 |
|------|---------|-----------|
| 整体规划成功率 | ≥ 90% | ≥ 95% |
| 意图分类准确率 | ≥ 85% | ≥ 95% |
| 任务解析精确匹配率 | ≥ 70% | ≥ 85% |
| 初次规划成功率 | ≥ 60% | ≥ 75% |
| 冲突解决率 | ≥ 80% | ≥ 90% |
| 平均端到端延迟 (3台) | ≤ 5s | ≤ 2s |
| 无效输入正确拒绝率 | ≥ 95% | ≥ 99% |

## 五、关键设计决策

### 5.1 数据集 vs 随机生成

| 方式 | 优点 | 缺点 | 本项目选择 |
|------|------|------|-----------|
| **固定数据集** | 可复现、可对比、可回归检测 | 覆盖有限 | ✅ **主要方式** |
| **随机生成** | 覆盖广 | 不可复现、结果不稳定 | ❌ AGENTS.md 已排除 |

### 5.2 评测粒度

```
批次级         → 整个调度请求是否成功
  任务级       → 每个机器人是否成功
    步骤级     → 每个时间步是否合法（检测结果）

建议：评测报告同时输出批次级 + 任务级汇总
```

### 5.3 LLM 输出评测的特殊性

NL 理解评测有两个挑战：

**挑战 1：LLM 输出不是确定性的**

解决方式：
```python
# 不要求精确匹配，使用容错匹配
def soft_match(actual: dict, expected: dict) -> float:
    """返回 0.0 ~ 1.0 的匹配度"""
    score = 0.0
    if actual["robot_id"] == expected["robot_id"]:
        score += 0.4  # 机器人 ID 匹配
    if actual["goal_location_id"] == expected["goal_location_id"]:
        score += 0.4  # 目标匹配
    if abs(actual.get("priority", 1) - expected.get("priority", 1)) <= 1:
        score += 0.2  # 优先级近似
    return score
```

**挑战 2：需要区分 LLM 错误 vs 系统错误**

```python
def classify_error(state) -> str:
    """区分错误来源。"""
    if state.errors and "LLM" in str(state.errors):
        return "llm_error"
    if state.errors and "A*" in str(state.errors):
        return "planning_error"
    if state.current_conflicts:
        return "conflict_error"
    return "unknown"
```

### 5.4 评测耗时控制

评测数据集 200 条，每条含 1 次 LLM 调用（~2s）+ A* 规划（~50ms）：

```
200 条 × (2s + 0.05s) ≈ 410s ≈ 7 分钟
```

建议：
- 完整评测：每周运行一次（全量）
- 快速评测（仅 NL 理解）：每次 prompt 修改后运行（子集）
- 增量评测：仅运行变更影响的场景

### 5.5 基线管理

```python
# 每次评测后保存基线
report = run_evaluation("eval/datasets/full")
save_baseline(report, "eval/baseline/latest.json")

# 下次运行后对比
latest = run_evaluation("eval/datasets/full")
baseline = load_baseline("eval/baseline/latest.json")
diff = compare_baselines(latest, baseline)

# 退化检测
if diff["planning_success_rate"] < -0.05:
    alert("⚠️ 规划成功率下降超过 5%！")
```

## 七、总结

| 维度 | 评测内容 | 数据集 | 关键指标 |
|------|---------|--------|---------|
| NL 理解 | 意图分类 + 任务解析 | nlu_test.json (100-200条) | 准确率、F1 |
| 路径规划 | A* + 冲突解决 + 重规划 | scenario_test.json (30-50个) | 成功率、重规划次数 |
| 端到端 | 完整流程 | e2e_test.json (50-100条) | 全部 checkpoints |
| 性能 | LLM/A* 延迟 | 上述所有 | P50/P95 延迟 |
| 鲁棒性 | 边界情况 | edge_cases.json | 正确拒绝/降级率 |

**核心原则**：
- 数据集是**固定的**（不随机生成），保证评测可复现
- LLM 输出用**容错匹配**（软匹配分数），不用精确字符串比较
- 每次评测**保存基线**，下次运行自动对比退化
- 与 AGENTS.md 保持一致：规划成功率按任务数计算，不引入随机场景评估
