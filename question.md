## 1. 项目宏观理解

### Q1.1 请用一句话介绍你的项目

**回答：**

> 这是一个基于 LangGraph 编排的智能仓储机器人调度系统，通过 DeepSeek 大模型理解自然语言指令，结合确定性 A* 路径规划和冲突检测算法，为 3~5 台仓储机器人生成无碰撞的时空路径，支持最多 3 次重规划和部分执行方案。

### Q1.2 为什么要做"LLM 理解 + 算法求解"这种分层？直接让大模型生成路径不行吗？

**追问意图：** 考察你对 LLM 能力边界的理解。

**回答：**

这个分层决策是本项目最核心的设计原则，原因有三：

1. **可验证性（最关键）。** 大模型生成的路径可能存在穿越障碍、违反物理约束、时间不连续等问题，且难以穷举验证。而 A* 搜索得到的路径在数学上保证最优性（给定启发函数），冲突检测器可以在多项式时间内完成全量验证。**"你不应该让模型猜路径，而应该让模型做它擅长的事——理解语义和做策略决策。"**

2. **可解释性与可审计性。** 当规划失败时，确定性模块可以精确报告"在 t=5 时刻，(3,4) 位置发生顶点冲突，涉及 R1 和 R2"；而纯 LLM 生成路径只能给出模糊的失败解释。

3. **安全边界。** 在仓储场景中，路径错误可能导致机器人碰撞或货架损坏。确定性算法提供了一层硬性的安全保障，LLM 的输出始终经过 Schema 校验 + 路径验证两层安检。

**类比：** 就像自动驾驶中，感知模块可以用大模型，但规划控制模块必须用确定性算法保证安全。

### Q1.3 项目中 LLM 和确定性代码的分界线具体在哪里？

**回答：**

```
LLM/Agent 负责：                        确定性模块负责：
├── 理解自然语言指令                    ├── A* 路径搜索
├── 提取机器人 ID / 目标位置 / 优先级    ├── 障碍物判断
├── 解析临时封闭约束                    ├── 时空预留表
├── 根据冲突报告选择重规划策略          ├── 冲突检测（顶点/交换/起点/终点）
├── 输出可解释说明                      ├── 路径验证（连续性/边界/障碍/安全）
                                         ├── 指标统计（成功率/耗时/展开节点数）
```

**关键规则：** LLM 永远不直接生成 `(x, y, t)` 路径坐标，只输出结构化任务描述（JSON Schema 约束）。

## 2. LLM Agent 架构与设计

### Q2.1 TaskParserAgent 的 Prompt 是如何设计的？有哪些关键优化点？

**回答：**

Prompt 设计采用**结构化模板 + 动态注入上下文**的策略：

```python
PARSE_SYSTEM_PROMPT = """你是一个仓储机器人任务解析器...
## 输出格式（强制 JSON Schema）
## 规则（6 条硬约束）
## 可用位置（动态注入：{locations_info}）
## 可用通道（动态注入：{corridors_info}）
## 可用机器人（动态注入：{robots_info}）
"""
```

关键优化点：

1. **少样本约束代替少样本示例。** 不提供完整示例 JSON，而是用 6 条规则约束 LLM 行为，因为少样本示例可能让模型过度拟合格式而非理解语义。

2. **上下文注入而非硬编码。** 地图的位置、通道、机器人列表是动态注入的，这样更换地图时无需修改 Prompt。

3. **输出 Schema 优先。** 明确要求返回纯 JSON，并在规则中定义每个字段的填充规则（如 "start 为 null 时由系统补全"，"priority 数字越小优先级越高"）。

4. **错误容忍。** `parse_warnings` 和 `parse_errors` 两个字段分离，让 LLM 遇到模糊指令时可以选择"给警告继续"而非"直接崩溃"。

5. **Temperature = 0.1。** 压制 LLM 的创造性，倾向于确定性输出。

### Q2.2 如果 LLM 返回的 JSON 格式错误或字段缺失，系统怎么处理？

**回答：**

系统有多层容错机制：

1. **JSON 提取层 (`_extract_json`)**：LLM 输出可能包裹在 ` ```json ... ``` ` Markdown 代码块中，也可能是纯文本中嵌入 JSON。系统先用正则 `r'\{[\s\S]*\}'` 尝试提取，再执行 `json.loads()`。

2. **字段补全层 (`_build_batch`)**：对于缺失的 `start` 字段，系统自动从 `RobotRegistry` 读取运行时位置；对于缺失的 `priority`，按任务出现顺序分配；对于重复的 priority，自动递增去重。

3. **校验失败返回结构化错误**：如果 LLM 调用本身失败（网络超时、API 密钥错误），返回 `TaskBatch(parse_errors=[...])`，整个流程标记为 `INFEASIBLE`。

4. **严格区分"LLM 解析错误"和"业务校验错误"**：前者标记为 `parse_errors`，后者在后续的 `validate_and_resolve_goals` 节点处理。

### Q2.3 如果用户输入的机器人名称在地图中不存在，系统如何响应？

**回答：**

系统遵循"**不猜测缺失坐标**"原则：

1. TaskParserAgent 在 Prompt 中注入了可用机器人列表（`robots_info`），LLM 应该只引用存在的机器人。
2. 如果 LLM 仍然输出了未知机器人 ID，`_build_batch` 方法会在解析时检查 `robot_registry.get_position(robot_id)`，如果返回 None，则添加解析错误并跳过该任务。
3. 如果 `parse_errors` 非空，`TaskBatch.is_valid` 返回 False，整个批次被标记为 `INFEASIBLE`。
4. 最终用户会收到结构化错误信息："Robot R5: unknown and no start specified"。

**核心设计哲学：** 宁可返回错误，也不让模型编造坐标。这在面试中是一个很好的"安全合规意识"体现。

### Q2.4 为什么不直接用 Function Calling 来替代手写 Prompt 解析？

**回答：**

好问题。我的考量如下：

1. **本项目的 LLM 调用是 OpenAI 兼容协议的 DeepSeek Chat**，DeepSeek 的 Function Calling 能力在中文场景下不如 GPT-4 稳定。使用手写 Prompt + JSON Schema 约束可以获得更可控的结果。

2. **结构化输出的可审计性。** 纯 JSON 输出可以被直接记录、审计、重放，而 Function Calling 的 schema 和执行日志在不同平台间格式不一。

3. **迁移灵活性。** 如果将来切换模型供应商（如切换到通义千问 Qwen），只需要修改客户端配置和 Prompt 中的少数描述，不需要重写 Function Calling schema。

4. **当然，如果明确使用 GPT-4 或支持可靠 Function Calling 的模型，我会改用工具调用方式，因为它的结构化输出更稳定。**

### Q2.5 ReplanningAgent 的决策逻辑是什么？它是基于规则还是基于 LLM？

**回答：**

ReplanningAgent **完全是基于规则的确定性代码，不调用 LLM**。这是"LLM 理解，算法求解"原则的延伸。

三层决策策略（按 retry_count 递增）：

| 重试次数 | 策略名称 | 具体操作 | 适用场景 |
|---------|---------|---------|---------|
| 0 (首次) | `fix_high_priority` | 固定高优先级机器人路径，为低优先级机器人重规划 | 首次冲突通常是优先级安排不当 |
| 1 | `add_time_window` | 在冲突点周围增加时间窗口约束（±2~3 步），强制等待 | 多机争夺同一空间区域 |
| 2 | `adjust_priority` | 交换冲突双方的优先级，重新规划 | 优先级顺序本身就是问题根源 |

**为什么不用 LLM 来做重规划决策？**

- 重规划决策是确定性的、可枚举的逻辑，用规则引擎更可靠、更快（微秒级 vs 秒级）。
- 路径规划失败的诊断需要精确的时间和位置信息，LLM 处理数字精度不够。
- 面试中可以说：**"我们要把 LLM 用在它最擅长的地方——模糊语义理解，而不是精确的算法决策。"**

## 3. LangGraph 编排引擎

### Q3.1 为什么选择 LangGraph 而不是直接写一个 Pipeline 或 State Machine？

**回答：**

1. **显式的有向图结构。** LangGraph 的 StateGraph 让整个调度流程可视化为一个 DAG，每个节点是纯函数，每条边是显式的条件路由。相比于隐式的 Pipeline（如按顺序调用函数列表），这种结构在新增/调整流程节点时更安全。

2. **内建的循环支持。** 重规划需要循环（最多 3 次），传统 Pipeline 要么通过递归调用实现（容易栈溢出），要么通过 while 循环加条件判断（状态管理混乱）。LangGraph 的 `add_conditional_edges` 天然支持带循环的流式编排。

3. **State reducer 模式。** `GraphState` 使用 TypedDict + `operator.add` reducer 机制，累积字段（如 `replan_history`、`errors`、`warnings`）自动 append，覆盖字段自动替换，状态管理零心智负担。

4. **可编译性和可测试性。** 编译后的 Graph 可以独立测试每个节点函数的输入输出，不需要模拟整个环境。

### Q3.2 GraphState 是如何设计的？accumulating 和 overwrite 字段怎么区分？

**回答：**

```python
class GraphState(TypedDict, total=False):
    request_id: str                    # 覆盖（每个请求唯一）
    original_instruction: str          # 覆盖
    task_batch: Optional[TaskBatch]    # 覆盖
    priority_order: list[str]          # 覆盖
    static_obstacles: set              # 覆盖
    dynamic_obstacles: set             # 覆盖
    initial_paths: dict                # 覆盖（初始规划结果）
    current_paths: dict                # 覆盖（当前工作路径）
    initial_conflicts: list[Conflict]  # 覆盖
    current_conflicts: list[Conflict]  # 覆盖
    retry_count: int                   # 覆盖（每次 +1）
    status: BatchStatus                # 覆盖（状态机流转）
    metrics: Optional[PlanningMetrics] # 覆盖
    # --- 以下为累积字段 ---
    replan_history: Annotated[list, operator.add]   # append-only
    task_results: Annotated[list, operator.add]     # append-only
    warnings: Annotated[list, operator.add]         # append-only
    errors: Annotated[list, operator.add]           # append-only
```

**设计原则：**

- **覆盖字段**对应于"当前快照"——最后一次写入就是最新值。
- **累积字段**对应于"历史日志"——每次写入都追加到列表末尾。
- 这种设计让每个节点函数只关注自己需要更新的字段，不需要手动维护其他字段的状态。

### Q3.3 图中的条件路由是如何工作的？画一下完整的流程拓扑。

**回答：**

完整的状态图拓扑（10 个节点，4 条条件边，1 条循环边）：

```
START
  │
  ▼
parse_instruction          # Node 1: LLM 任务解析
  │
  ▼
validate_and_resolve_goals # Node 2: 目标入口分配
  │
  ▼
build_obstacles            # Node 3: 构建障碍物集合
  │
  ▼
initial_plan               # Node 4: 初始独立 A* 规划
  │
  ▼
conflict_check  ─────────────────────────────►  route_after_conflict_check
  │                                                │
  ├─ 无冲突 ──────────────────── "validate" ──────► validate_final
  ├─ 有冲突 & retry<3 ───────── "replan" ─────────► replan_decide
  │                                                    │
  │                                                    ▼
  │                                               apply_replan
  │                                                    │
  └────────────────────────── "replan" ◄──────── conflict_check (循环)
  │
  ├─ 全部失败或 retry>=3 ──── "partial" ─────────► partial_execution
  │                                                    │
  │                                                    ▼
  │                                              route_after_partial_execution
  │                                               ├─ "validate" → validate_final
  │                                               └─ "infeasible" → compute_metrics
  │
  └─ 硬不可行 ──────────── "infeasible" ─────────► compute_metrics
                                                       │
                                                       ▼
                                                  validate_final
                                                       │
                                                       ▼
                                                  route_after_validate
                                                   ├─ "compute" → compute_metrics
                                                   └─ "end" → compute_metrics
                                                       │
                                                       ▼
                                                       END
```

**关键条件路由函数 `route_after_conflict_check` 的逻辑：**

```python
if status == INFEASIBLE:          → "infeasible"
if no_conflicts and all_success:  → "validate"
if all_failed:                    → "partial"
if conflicts and retry < 3:       → "replan"
else:                             → "partial"
```

## 4. 确定性算法：A* 与多机器人路径规划

### Q4.1 你的 A* 和标准 A* 有什么不同？为什么需要"时空 A*"？

**回答：**

**核心区别：状态空间从 `(x, y)` 扩展到 `(x, y, t)`。**

标准 A* 只关心空间位置，而仓储场景需要：
1. **等待动作。** 机器人可以原地等待让行，这是二维 A* 无法表达的（二维 A* 到达目标就结束）。
2. **时间维度的障碍。** 临时封闭通道、其他机器人的预留路径都需要"在 t 时刻某格不可通行"这种细粒度约束。
3. **终点持续占用。** 机器人到达目标后持续占用该格，阻止其他机器人后续进入。

**实现细节：**

```python
# 状态: (x, y, t)
# 动作空间: [(0,-1), (0,1), (-1,0), (1,0), (0,0)]  # 上下左右 + 等待
# g-cost: 累积移动/等待代价
# h-cost: Manhattan 距离（到目标）
# 约束: is_free(x, y, t) 检查静态障碍、动态障碍、预留表
```

时空 A* 的代价 vs 传统 A*：
- 状态空间大了 `max_timestep` 倍（默认 200），所以必须设置 `max_timestep` 防止无限搜索。
- 使用 `robot_id` 参数豁免自身预留（同一台机器人的预留不阻塞自己）。
- 边安全检测（`is_move_safe`）预防交换冲突。

### Q4.2 A* 的启发函数为什么选择 Manhattan 距离？有没有考虑过其他启发函数？

**回答：**

1. **可采纳性（Admissibility）。** Manhattan 距离是对网格四方向移动的低估，满足 A* 可采纳性要求，保证找到最优路径。欧几里得距离在某些网格布局下可能过度低估。

2. **计算效率。** Manhattan 距离是 O(1) 计算，而更复杂的启发函数（如对角线距离、模式数据库）虽然可能减少展开节点数，但计算开销更大。

3. **场景适配。** 我们允许移动代价和等待代价相同（均为 1.0），Manhattan 距离恰好是最优路径代价的下界。如果将来引入非均匀代价，我可能会改用 weighted A* 或考虑地形信息的启发函数。

4. **潜在改进方向：** 在大型地图上（>50×50），可以引入层次化 A* 或跳点搜索（JPS）来减少展开节点数。

### Q4.3 初始规划和重规划时 A* 的执行有什么区别？

**回答：**

| 特性 | 初始规划 | 重规划 |
|------|---------|--------|
| **预留表** | 不使用（独立规划） | 使用（高优先级机器人路径已预留） |
| **动态障碍** | 仅静态障碍 + 临时封闭 | 静态障碍 + 临时封闭 + 冲突窗口约束 |
| **规划顺序** | 任务顺序（无优先级） | 按优先级顺序（高优先级先规划） |
| **目的** | 检测首次冲突 | 通过预留避免冲突 |

**初始规划不使用预留表**是刻意设计——如果不冲突的路径在留有预留表的情况下本来就能规划成功，但如果我们一上来就用预留表，就会把"优先级规划导致的不可行"和"地图本身导致的不可行"混在一起，不利于冲突诊断。

### Q4.4 为什么是优先级规划而不是 CBS？优先级规划有什么已知的缺陷？

**回答：**

**当前 MVP 选择优先级规划的原因：**

1. **机器人规模小（3~5 台）。** 在有限数量下，通过最多 3 次优先级调整和重规划，大多数冲突都可以解决。
2. **实现复杂度低。** 优先级规划 + 预留表的实现不到 200 行，而完整的 CBS（Conflict-Based Search）需要维护约束树、高层搜索和低层搜索，复杂度大增。
3. **延迟满足。** AGENTS.md 已经明确将 CBS/ECBS 标记为后续候选方案。

**优先级规划的已知缺陷（面试重点！）：**

- **不完备性（Incompleteness）。** 即使存在可行解，也可能因为优先级顺序不当而找不到。例如：两机器人相向而行，如果低优先级机器人被要求让行但空间不足，高优先级机器人也被阻塞。
- **次优性。** 即使找到解，也可能不是全局最优（总路径和或总时间最长）。
- **公平性问题。** 低优先级机器人总是被要求绕行或等待。

**本项目的缓解措施：**
- 最多 3 次重规划 + 策略调整（时间窗口约束、优先级交换）。
- 部分执行机制（至少让部分机器人先完成任务）。
- 后续规划：当机器人规模扩展或求解完备性要求提升时，替换为 CBS/ECBS。

## 5. 冲突检测与重规划策略

### Q5.1 你们检测哪几类冲突？为什么边冲突（edge conflict）和交换冲突（swap conflict）是分开的？

**回答：**

检测 4 类冲突：

| 冲突类型 | 定义 | 示例 |
|---------|------|------|
| **起点冲突 (START)** | t=0 时两台机器人占据同一格 | R1 和 R2 都从 (0,0) 出发 |
| **终点冲突 (GOAL)** | 两台机器人终点相同，或后到达者要进入已占用的终点 | R1 已停在 (8,2)，R2 也要进入 (8,2) |
| **顶点冲突 (VERTEX)** | 同一时间在同一格子 | R1 在 t=5 时刻在 (3,4)，R2 也在 |
| **交换冲突 (SWAP)** | 两机器人交换相邻位置 | R1: (0,0)→(1,0)，R2: (1,0)→(0,0) |

**为什么 SWAP 要和 VERTEX 分开？**

交换冲突是顶点冲突的**时空交集但非共时**——两台机器人永远不会同时占据同一格，但它们的路径交叉了。具体来说：
- 在 t 时刻：R1 在 A，R2 在 B（A ≠ B）→ 顶点检测发现不了。
- 在 t+1 时刻：R1 在 B，R2 在 A → 顶点检测同样发现不了。

但这两台机器人实际上是**交换了位置**，这在物理上意味着它们"穿过"了对方，在单格同构机器人场景下是不可行的。所以要单独检测。

### Q5.2 冲突检测中"终点持续占用"是怎么处理的？

**回答：**

这是多机器人路径规划中的关键细节。当机器人到达终点后，它不会消失，而是持续占用该格子直到调度结束。

实现上采用**路径补齐（padding）策略**：

```python
# 在 ConflictDetector.detect() 中：
last_pos = (rp.path[-1].x, rp.path[-1].y)
last_t = rp.path[-1].time
# 将终点位置扩展到 max_t + 10
for t in range(last_t + 1, max_t + 1):
    padded[rid][t] = last_pos
```

这样，如果 R1 在 t=2 到达 (3,4)，而 R2 在 t=5 要进入 (3,4)，冲突检测器会在 t=5 发现顶点冲突。

**时间窗口选择 `max_t + 10` 而非 `max_t`**：这是为了防止规划时域的边界效应，给终点预留一点额外的时间窗口。

### Q5.3 三次重规划的策略为什么这样设计？有没有可能更多次？

**回答：**

三次策略的设计思想是**从"最小侵入"到"最大调整"的递进**：

1. **第 1 次（fix_high_priority）**：最小改动。保留高优先级的路径不变，只对低优先级机器人重规划。这是"保重点"策略。

2. **第 2 次（add_time_window）**：增加时间约束。在冲突点周围设置禁行时间窗口（±2~3 步），强制机器人提前等待或绕行。这是"柔性避让"。

3. **第 3 次（adjust_priority）**：调整优先级。如果前两次都失败，说明当前的优先级顺序本身就是问题根源，需要交换冲突双方的优先级别。这是"颠覆性调整"。

**为什么不多于 3 次？**

- AGENTS.md 明确约定了最多 3 次（业务规则第 20 条）。
- 对于 3~5 台机器人，3 次已经覆盖了所有合理的调整方向。更多次重试大概率是无效计算。
- 防止无限循环导致死锁（但 LangGraph 的循环边理论上没有次数限制，所以要靠 `retry_count` 和条件路由控制）。

**面试中可以说：** "3 次的设计是基于对 3~5 台机器人场景的经验判断，如果验证中发现某类场景需要更多重试，可以调整 `MAX_RETRIES` 常量。"

### Q5.4 部分执行是怎么实现的？如何确保删除一台机器人后不导致新的冲突？

**回答：**

部分执行遵循"**安全优先于任务数量**"原则：

**算法流程：**
1. 收集所有失败/冲突的机器人 ID。
2. 按优先级从低到高依次删除冲突机器人（优先保留高优先级任务）。
3. 对于每个候选子集，执行 `_plan_subset()` 函数：
   - 被删除的机器人不消失，其**起点位置作为全时段动态障碍物**加入约束。
   - 对保留的任务按优先级顺序，使用预留表重新规划。
   - 检查最终路径是否还有冲突。
4. 返回第一个可行的子集。

**关键安全机制（`_plan_subset`）：**
```python
# 未执行机器人作为全时段障碍
for pos in occupied_starts.values():
    for t in range(max_t + 1):
        combined_dynamic.add((pos[0], pos[1], t))
# 重新规划 + 冲突检测
detector = ConflictDetector()
if detector.detect(results):  # 有残留冲突 → 失败
    return None
```

**为什么要重新规划而不是复用旧路径？**
- 删除一台机器人后，原来为避让它而绕行的其他机器人路径可能不再最优。
- 未执行机器人的起点作为新障碍，会影响其他机器人的路径。
- 这就是 AGENTS.md 中要求的 "不得通过简单删除冲突任务后直接复用旧路径"。

## 6. 数据模型与状态管理

### Q6.1 你们的 BatchStatus 状态机是怎么设计的？为什么需要这么多中间状态？

**回答：**

```python
class BatchStatus(str, Enum):
    RECEIVED = "received"
    PARSED = "parsed"
    VALIDATED = "validated"
    INITIAL_PLANNED = "initial_planned"
    CONFLICT_CHECKED = "conflict_checked"
    REPLANNING = "replanning"
    SUCCEEDED = "succeeded"
    PARTIALLY_SUCCEEDED = "partially_succeeded"
    INFEASIBLE = "infeasible"
```

**设计理念：** "可观测的流程状态 = 可调试的系统。"

每个中间状态对应一个明确的流程节点，带来的好处：
1. **精确的错误定位。** 如果返回 `PARSED` 但后续状态丢失，说明问题出在 validation 阶段。
2. **指标拆解。** 可以分别统计解析耗时、初始规划耗时、重规划耗时。
3. **LLM 调用可审计。** 可追踪到哪个阶段调用了 LLM、调用了多少次。

对比"简单三种状态（成功/失败/进行中）"的方案，我们的设计更有利于排查多机规划中复杂的冲突问题。

### Q6.2 地图模型中为什么要把 facility_cells 和 entry_cells 分开？

**回答：**

这是一个**领域驱动的设计决策**，反映了仓储场景的真实物理约束：

- **facility_cells（设施本体格）**：装卸台、充电桩等设施的物理位置，机器人不能"停在上面"。这些格子可能是障碍物（从路径角度看）或不可达区域。
- **entry_cells（入口格）**：机器人实际可以停靠、等待、执行任务的格子。一个设施可以有多个入口（如装卸区南北两侧都有入口）。

**设计价值：**
1. **语义清晰。** LLM 解析时知道"去充电区"不是停在充电桩上，而是停在旁边的入口。
2. **多入口选择。** `LocationResolver.select_best_goal()` 可以根据曼哈顿距离和占用情况选择最优入口。
3. **容量管理。** `capacity` 字段检查入口格数量是否足够，防止多台机器人分配到同一个人口。

### Q6.3 如果两个机器人同时要去同一个多入口设施，系统如何分配入口？

**回答：**

通过 `LocationResolver.select_best_goal()` 方法实现：

```python
def select_best_goal(self, identifier, start, occupied):
    free = self.get_free_entry_cells(identifier, occupied)  # 排除已占用的
    if not free:
        return None  # 所有入口已满
    # 按曼哈顿距离排序，选最近的
    free.sort(key=lambda g: abs(g[0]-start[0]) + abs(g[1]-start[1]))
    return free[0]
```

分配流程在 `validate_and_resolve_goals` 节点中按优先级顺序进行：
1. 高优先级机器人先选，占用一个入口。
2. `occupied_goals` 集合记录已分配的入口。
3. 低优先级机器人从剩余入口中选择。

**缺陷：** 贪心策略不一定全局最优。例如：高优先级机器人选了"最近"但"唯一"的入口，导致低优先级机器人只能绕远路。但 3~5 台场景下，贪心策略的次优性影响很小。

### Q7.2 FastAPI 的接口设计是怎么样的？如何保证并发安全？

**回答：**

API 端点设计：

```
POST /api/schedule       → 执行调度（支持 NL 或结构化任务）
GET  /api/map            → 获取地图
PUT  /api/map            → 更新地图并重新加载
GET  /api/runtime        → 获取运行时状态
PUT  /api/runtime        → 更新运行时状态并重新加载
```

3. **UUID 请求 ID。** 每个请求分配唯一 `request_id`，方便日志追踪和问题定位。

**面试可以补充：** "当前 MVP 没有使用数据库或消息队列，所以并发模型是简单的请求-响应。如果将来引入执行状态回传，我会考虑用 Redis 存储运行时状态，并引入乐观锁来处理并发位置更新。"

   ```python
   try:
   ```

   如果 API Key 配置错误或额度耗尽，用户仍然可以使用结构化模式完成全流程规划。

3. **面试可以提及的改进方向：** "如果生产环境要求高可用，我会引入重试机制（指数退避）、模型降级（切换到备用模型如通义千问）、以及缓存策略（对相似的 NL 指令缓存解析结果）。"

## 8. 场景扩展与开放问题

### Q8.1 如果要把机器人数量从 3~5 台扩展到 50 台，系统需要做哪些改造？

**回答：**

这是一个非常好的考察系统设计扩展性的问题。需要从多个层面改造：

1. **路径规划算法升级（最关键）。** 优先级规划在 50 台场景下几乎不可用（不完备性问题急剧放大）。需要替换为：
   - **CBS/ECBS**：Conflict-Based Search，通过约束树（Constraint Tree）搜索无冲突解。
   - **LRA* / WHCA***：适用于大规模多智能体路径规划的窗口式算法。
   - **分而治之**：将 50 台机器人按空间区域分组，每组独立规划后协调边界。

2. **搜索空间优化。** 当前 Manhattan 距离 + 完整时空 A* 在 50 台 × 20×20 地图上状态空间爆炸。需要引入：
   - 带时间窗口的 A*（限制最大规划 horizon）。
   - 跳点搜索（JPS）减少展开节点数。
   - 层次化路径规划（先粗粒度路径，再细粒度插值）。

3. **部分执行策略升级。** 当前枚举子集（O(2^n)）在 50 台时不可行。需要：
   - 贪心选择（优先保留高优先级/短路径的任务）。
   - 整数规划或约束规划求解最优子集。

4. **性能优化。** Python 在 50 台场景下可能成为瓶颈。可以考虑：
   - A* 核心用 Cython 或 Rust 重写。
   - 并行化：多台机器人的路径规划可并行执行。
   - 增量重规划：只重新规划受影响的机器人，而非全量。

### Q8.2 如果要支持"不同机器人有不同的速度"，A* 和冲突检测需要怎么改？

**回答：**

这是很好的系统灵活性考题。改动涉及多个模块：

**A* 规划器：**
- 将 `speed` 参数化，每个机器人有自己的 `move_cost`（速度越快，cost 越低）和时间步缩放。
- 状态转换改为：移动一步的时间步长 = `1/speed`（浮点数），而非当前固定的 +1。
- 启发函数需要调整为 `Manhattan / speed` 以保持可采纳性。

**冲突检测：**
- 当前假设"每时间步所有机器人同步移动一格"，不同速度下需要改为**连续时间冲突检测**：
  - 计算每台机器人的时空轨迹为分段线性函数 `(x, y, t)`。
  - 检测两条轨迹是否存在交集（时间 + 空间）。
  - 处理"快机器人超过慢机器人"的场景（新的冲突类型）。

**预留表：**
- 预留的粒度从离散时间步改为连续时间区间 `(x, y, [t_start, t_end])`。
- 边预留需要更复杂的碰撞检测。

**面试亮点：** "我会在 `MovementRules` 模型中添加 `speed` 字段（默认 1.0），然后在 `AStarPlanner` 的 `plan()` 方法中加入 `robot_speed` 参数，让规划框架本身就支持异步步长，而不是在外部做后处理适配。"

### Q8.3 如果要引入"在线动态重调度"（机器人执行中遇到新障碍），系统架构需要做什么变化？

**回答：**

这是一个从"静态规划"到"动态执行"的架构升级：

1. **执行状态机引入。** 需要新增 `EXECUTING`、`PAUSED`、`COMPLETED`、`FAILED` 等执行状态，计划在 `planning_state.py` 中新增 `ExecutionStatus` 枚举。

2. **状态回传通道。** 当前系统是"规划-输出"的单向流程，动态调度需要双向通信：
   - 机器人定期回传当前位置和状态（通过 MQTT / WebSocket / gRPC）。
   - 系统检测到"实际路径偏移"或"新障碍物报告"后触发局部重规划。

3. **增量重规划而非全量重规划。** 动态调度不需要重新规划所有机器人，只需要：
   - 确定受影响的机器人（在新障碍物路径上或即将进入的机器人）。
   - 固定不受影响机器人的路径。
   - 对受影响的机器人执行局部重规划（D* Lite 或局部时空 A*）。

4. **D* Lite 算法。** 专门用于动态环境下的增量路径搜索，能在已知环境变化时高效更新路径，不需要从头搜索。

5. **一致性保障。** 引入"规划版本号"机制，确保机器人不会同时收到冲突的路径更新指令。

**面试亮点：** "我会参考 ROS 2 的导航框架设计思路，将系统分为全局规划器（global_planner，低频，算最优路径）和局部规划器（local_planner，高频，做避障和路径跟踪），两者通过 costmap 共享障碍信息。"

### Q8.4 这个项目如果要在阿里云上部署到生产环境，你会怎么做？

**回答：**

**架构方案：**

```
[Client/NL Input] → [SLB] → [FastAPI Pods] → [Redis Cache] → [MySQL/PostgreSQL]
                        ↓                          ↑
                   [Celery Task Queue] ─────→ [Asyncio Workers]
                        ↓
                   [LLM Service (DeepSeek/Qwen)]
```

**具体设计：**

1. **容器化。** Docker + Kubernetes (ACK)，FastAPI 服务水平伸缩 2~4 个 Pod。
2. **异步处理。** 长时间规划任务通过 Celery 异步执行，客户端轮询结果。
3. **缓存。** 相同/相似的 NL 指令缓存解析结果（Redis），减少 LLM 调用成本。
4. **数据持久化。** 地图配置存入 MySQL，规划结果和指标存入数据库供后续分析。
5. **可观测性。** 集成阿里云 SLS（日志服务）、ARMS（APM 监控），关键指标：规划成功率、LLM 调用延迟、A* 展开节点数 P99 分布。
6. **模型服务。** 如果使用通义千问，可以通过阿里云 DAS（模型服务平台）部署，享受内网低延迟。
