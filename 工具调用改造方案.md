# 工具调用改造方案：从 LLM 路由到 LLM 自主选择工具

> 参考 Hermes Agent 工具调用系统，设计适用于本项目的轻量级工具调用方案。
> 不修改现有代码，仅分析并给出设计方案。

---

## 一、当前架构：LLM 做路由，代码执行

### 1.1 当前交互流程

```
用户: "R1去装卸区"
        │
        ▼
  ┌─────────────────────┐
  │  classify_intent()  │  ← LLM 调用（固定 tool_choice）
  │  (Function Calling) │     代码硬编码调用 classify_intent 工具
  └─────────┬───────────┘      LLM 只负责填参数
            │
            ▼  {"intent": "schedule"}
  ┌─────────────────────┐
  │  if intent ==       │  ← Python 代码做路由
  │    "schedule"       │     12 路 if/elif 分支
  │    "cron_create"    │
  │    "cron_list"      │
  │    ...              │
  └─────────┬───────────┘
            │
            ▼  (schedule 分支)
  ┌─────────────────────┐
  │  parse()            │  ← 另一个 LLM 调用（固定 tool_choice）
  │  (Function Calling) │     代码硬编码调用 parse_warehouse_tasks 工具
  └─────────┬───────────┘
            │
            ▼  TaskBatch
  ┌─────────────────────┐
  │    规划引擎          │
  └─────────────────────┘
```

### 1.2 当前的问题

| 问题 | 表现 |
|------|------|
| **LLM 没有选择权** | 代码固定 `tool_choice = {"function": {"name": "classify_intent"}}`，LLM 只能填参数 |
| **路由在代码中** | 12 路 `if/elif` 分支是硬编码的，新增意图需修改 Python 代码 |
| **两次 LLM 调用** | 先 classify_intent，再 parse，两次独立调用无法共享上下文 |
| **意图和解析分离** | 分类时 LLM 不知道具体调度参数，解析时 LLM 不知道意图上下文 |

## 二、Hermes 工具调用的核心模式

Hermes 的做法（简化）：

```
LLM API 调用
  ├── tools: [tool_A, tool_B, tool_C, ...]  ← 全部工具定义一次传入
  │
  └── LLM 返回 tool_calls: [                 ← LLM 自主选择调用哪个
        {name: "tool_B", args: {...}},
      ]
        │
        ▼
  registry.dispatch("tool_B", args)          ← 注册表分发执行
        │
        ▼
  结果返回 LLM，进入下一轮推理
```

**关键区别：**
- Hermes 的 LLM **自主选择**调用哪个工具（tool_choice = "auto"）
- 当前项目的 LLM **被动填充参数**（tool_choice = 固定工具名）
- Hermes 的工具是**无状态纯函数**，通过注册表统一管理
- 当前项目的"路由"是 Python if/elif，不是工具调用

## 三、改造方案：轻量级工具调用

### 3.1 总体设计

将当前的"LLM 分类 → 代码路由"模式，改为**"LLM 自主选择工具 → 注册表分发"**模式。

```
改造前：                         改造后：
                                    ┌──────────────────┐
                                    │  LLM 收到全部工具  │
                                    │  定义，自主选择调用  │
                                    └────────┬─────────┘
                                             │
┌──────────────┐     ┌────────────┐          ▼
│ classify_intent│────→  if/elif   │    ┌──────────────────┐
│ (固定调用)     │     │ (代码路由) │    │  schedule_tool    │ ← LLM 直接选
└──────────────┘     └─────┬──────┘    │  cron_create_tool │ ← LLM 直接选
                           │           │  cron_delete_tool │ ← LLM 直接选
                           ▼           │  qa_tool          │ ← LLM 直接选
                    ┌──────────────┐    │  show_map_tool   │ ← LLM 直接选
                    │ parse()      │    │  ...             │
                    │ (固定调用)    │    └────────┬─────────┘
                    └──────┬───────┘             │
                           ▼                     ▼
                    ┌──────────────┐    ┌──────────────────┐
                    │  规划引擎     │    │  注册表分发执行    │
                    └──────────────┘    └──────────────────┘
```

### 3.3 核心流程变化

#### 改造前（当前）

```
一轮 LLM 调用：classify_intent（固定）→ 返回意图标签
    ↓
十二路 if/elif（代码路由）
    ↓
（如果是 schedule）
二轮 LLM 调用：parse（固定）→ 返回结构化任务
    ↓
规划引擎
```

#### 改造后

```
一轮 LLM 调用：tools=[schedule, cron_create, qa, show_map, ...]（全部工具）
              tool_choice="auto"（LLM 自主选择）
    ↓
LLM 返回 tool_calls: [{name: "schedule_robots", args: {...}}]
    ↓
注册表 dispatch("schedule_robots", args)
    ├── LLM 已在 args 中填好了 robot_id、goal_location_id
    └── 直接调用规划引擎（无需二次 LLM 解析）
```

**核心变化：从两次 LLM 调用变为一次。** 因为 LLM 直接输出结构化调度参数，不再需要先分类再解析。

## 四、注册表设计（简化版）

不需要像 Hermes 那样复杂（810 行的 registry），本项目只需要一个轻量级的字典。

```python
# app/tools/registry.py

class ToolRegistry:
    """轻量级工具注册表。"""

    def __init__(self):
        self._tools: dict = {}  # name → ToolDef

    def register(self, name: str, description: str, parameters: dict, handler: callable):
        self._tools[name] = {
            "name": name,
            "description": description,
            "parameters": parameters,
            "handler": handler,
        }

    def get_openai_tools(self) -> list[dict]:
        """返回 OpenAI Function Calling 格式的工具列表。"""
        return [
            {
                "type": "function",
                "function": {
                    "name": t["name"],
                    "description": t["description"],
                    "parameters": t["parameters"],
                },
            }
            for t in self._tools.values()
        ]

    def dispatch(self, name: str, args: dict) -> str:
        """执行工具，返回结果字符串。"""
        tool = self._tools.get(name)
        if tool is None:
            return json.dumps({"success": False, "error": f"Unknown tool: {name}"})
        try:
            result = tool["handler"](args)
            return json.dumps({"success": True, "data": result})
        except Exception as e:
            return json.dumps({"success": False, "error": str(e)})
```

### 与 Hermes 的对比

| 特性 | Hermes | 本项目（简化版） |
|------|--------|----------------|
| 代码量 | ~810 行 | ~40 行 |
| 并发执行 | 线程池 + 并行/串行决策 | 单线程串行 |
| 工具发现 | AST 扫描 + 自动 import | 手动注册 |
| 护栏 | 插件 hook + 审批 + 安全检查 | 无（不需要） |
| 工具集 | 40+ toolset，递归组合 | 无（全部可用） |
| 中间件 | 参数转换 + 结果变换 | 无 |
| MCP 支持 | 完整 MCP 协议 | 无 |

## 五、改造效果对比

### 5.1 LLM 调用次数

```
改造前: classify_intent(1次) + parse(1次) = 2 次 LLM 调用/每次调度
改造后: 1 次 LLM 调用（LLM 直接选择 + 输出参数）

省去 50% 的 LLM 调用，同时省去 50% 的 API 延迟和费用。
```

### 5.2 代码复杂度

```
改造前:
  1. classify_intent() 方法          (80 行)
  2. 12 路 if/elif 分支（server.py）  (60 行)
  3. 12 路 if/elif 分支（message_handler.py）(50 行)
  4. parse() 方法                     (170 行)
  5. _build_batch() 方法              (80 行)
  总计: ~440 行用于"路由 + 解析"

改造后:
  1. 12 个工具定义（JSON Schema）      (60 行)
  2. 12 个 handler 函数                (120 行)
  3. ToolRegistry                      (40 行)
  4. 1 次 LLM 调用                     (20 行)
  总计: ~240 行用于"工具调用"
  
减少 ~45% 的代码量。
```

### 5.3 可扩展性

```
改造前: 添加新功能
  1. 在 classify_intent 的 intent 枚举中加新值
  2. 在 server.py 加 elif 分支
  3. 在 message_handler.py 加 elif 分支
  4. 实现处理函数
  修改 4 处代码

改造后: 添加新功能
  1. 定义新工具的 Schema
  2. 注册 handler
  修改 2 处代码（且 handler 可独立测试）
```

### 6.3 新的 LLM 调用

```python
# 改造后的一次调用

def handle_message(text: str) -> dict:
    """一次 LLM 调用替代 classify_intent + parse 两次调用。"""

    # 1. 获取所有工具定义
    tools = registry.get_openai_tools()  # 12 个工具定义

    # 2. 一次 LLM 调用，让 LLM 自主选择工具
    response = client.chat.completions.create(
        model=...,
        messages=[
            {"role": "system", "content": SYSTEM_PROMPT},
            {"role": "user", "content": text},
        ],
        tools=tools,
        tool_choice="auto",  # ← LLM 自主选择，不再是固定调用
    )

    # 3. 处理 LLM 返回的 tool_calls
    tool_call = response.choices[0].message.tool_calls[0]
    result = registry.dispatch(tool_call.function.name, tool_call.function.arguments)

    return result
```

## 七、风险与注意事项

### 7.1 LLM 选择错误工具的风险

**当前架构**：`classify_intent` 固定调用，LLM 只能填 intent 参数，即使填错了，也有代码的 if/elif 逻辑保底。

**改造后**：LLM 可能选择错误的工具（如把调度指令选了 `answer_question`）。

**缓解措施**：

```python
# 在 dispatch 中加一层后验证
def dispatch(self, name: str, args: dict) -> str:
    # 1. 参数 Schema 校验（pydantic）
    # 2. 如果调度工具的参数缺少 robot_id 或 goal_location_id
    #    返回明确错误，让 LLM 重试
    # 3. 对危险操作（delete_all、map_modify）加确认步骤
```

### 7.2 工具数量控制

当前 12 个工具可以一次全部发给 LLM。如果未来扩展到 30+，需要考虑按场景分组（参考 Hermes 的 toolset 系统）。

### 7.3 与现有规划引擎的解耦

`schedule_robots` 工具的 handler 需要将 LLM 输出的结构化数据直接转换为 `RobotTask`，然后调用规划引擎。这需要确保 handler 的输出格式与规划引擎的输入兼容。

## 八、总结

| 维度 | 当前（LLM 路由） | 改造后（工具调用） |
|------|-----------------|------------------|
| **LLM 调用次数** | 2 次（分类 + 解析） | 1 次 |
| **路由位置** | Python if/elif | LLM 自主选择 |
| **新增功能的改动点** | 改 4 处代码 | 注册 1 个新工具 |
| **代码量** | ~440 行 | ~240 行 |
| **LLM 灵活度** | 低（只能填参数） | 高（能选工具） |
| **调试难度** | 低（分支清晰） | 中（依赖 LLM 决策） |
| **对 Hermes 的参考度** | ❌ 完全不同的模式 | ✅ 采用注册表 + 分发模式 |

**核心变化一句话**：
> **把"LLM 填分类标签 → 代码路由"改为"LLM 选工具 → 注册表执行"，从 12 路 if/elif 变为 12 个工具定义，从 2 次 LLM 调用变为 1 次。**
