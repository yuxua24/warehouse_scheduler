# 智能仓储机器人调度系统

本文件面向参与本项目开发、维护、测试、重构和评审的 AI Coding Agent，用于约束后续实现不得偏离已经确认的产品方向、业务规则和技术边界。

## 1. 项目摘要

本项目是一个面向固定仓库网格地图的智能多机器人调度系统。系统接收自然语言任务指令，将其解析为结构化机器人任务，并调用确定性的路径规划与冲突检测工具，为 3 至 5 台同构机器人生成无碰撞的时空路径。

核心问题包括：

- 将自然语言中的机器人、起点、目标区域、优先级和临时封闭约束转换为可验证的结构化数据；
- 在固定纯网格地图上使用 A* 路径规划；
- 检测多机器人路径中的顶点冲突、交换冲突以及起点、终点相关冲突；
- 在发现冲突后分析原因，并通过优先级调整、等待、绕行和局部重规划最多重试 3 次；
- 当全部任务无法同时执行时，允许生成经过重新验证的部分执行方案；
- 输出路径规划成功率和规划耗时等运行指标。

目标用户尚未形成完整角色定义。当前可确认的主要使用场景是：仓储调度或运维人员通过自然语言下达多个机器人任务，并在同一批次中附带临时封闭通道等运行时约束。

当前阶段是顶层设计和接口约束已基本明确、具体代码库和技术栈尚未确认。当前最重要的开发目标是完成 MVP 的确定性规划内核、自然语言结构化解析、冲突诊断与最多 3 次重规划流程。

项目最终愿景是在规划能力之上接入机器人任务下发和执行状态回传，同时分别统计：

- 路径规划成功率；
- 实际任务完成率。

2. 本文件中标记为“已确认”的规则；
3. 现有经过测试且未与已确认规则冲突的代码行为；
4. 标记为“当前倾向”或“建议方案”的内容；
5. Agent 自身的实现假设。

后续明确修改的决定覆盖早期方案。已被删除、否决或替换的设计不得继续作为当前需求实现。

如果实际代码、任务描述和本文存在实质冲突，不得静默选择其中之一。Agent 必须：

- 明确记录冲突；
- 优先实施影响最小、可回退的方案；

## 3. 项目目标

### 3.1 总体目标

将自然语言仓储任务可靠地转换为结构化、多机器人、无碰撞的时空路径规划结果。

### 3.2 当前阶段目标

建立模块化、可扩展、可读且可测试的 MVP，包括：

- 固定 JSON 网格地图加载；
- 运行时机器人位置与临时封闭约束处理；
- 自然语言结构化任务解析；
- 时空 A*；
- 顶点冲突和交换冲突检测；
- 冲突诊断与最多 3 次重规划；
- 部分执行方案；
- 路径规划成功率和规划耗时等指标。

### 3.3 用户价值目标

- 用户可以用自然语言描述多机器人任务，而不必直接编写坐标和路径；
- 系统输出必须可验证、可解释，不依赖模型“猜测”路径；
- 发生冲突或不可达时，系统应给出明确的失败类型和重规划记录；
- 部分任务可行时，系统允许返回安全的部分执行方案，而不是简单地整批失败。

### 3.4 技术与工程目标

- LLM/Agent 负责语言理解和策略决策；
- A*、冲突检测、预留表和路径验证由确定性代码实现；
- Agent 与工具之间只交换结构化对象；
- 模块之间通过明确数据模型和接口解耦；
- 最终结果必须通过确定性验证后才能标记成功。

### 3.5 MVP 成功标准

MVP 至少应满足：

- 能从固定 JSON 地图加载地图结构；
- 能将符合支持范围的自然语言解析为结构化任务和临时封闭约束；
- 能为每个可执行任务生成带时间步的连续路径；
- 能检测顶点冲突和交换冲突；
- 能在冲突后最多重规划 3 次；
- 能输出全部成功、部分成功或不可行结果；
- 能计算路径规划成功率；
- 能记录总规划耗时和其他已实现的规划指标；
- 所有标记为成功的路径均通过地图、连续性、起终点和多机器人冲突验证。

## 4. 非目标与范围边界

### 4.1 MVP 明确不包含

- 真实机器人控制器或任务下发；
- 真实机器人执行状态回传；
- 实际任务完成率；
- 前端页面或可视化界面；
- 用户、管理员、登录、鉴权和权限系统；
- 数据库、缓存、消息队列、对象存储或搜索服务；
- 随机地图或随机场景生成器；
- 基于 100 组随机场景的批量评估模块；
- 场景级成功率、场景级初次冲突率或场景级重规划触发率；
- 对角线移动；
- 不同机器人尺寸、速度或运动能力；
- 转向成本；
- 多机器人并排宽度建模；
- 完整 CBS/ECBS 求解器；
- 动态在线重新调度和实时避障执行闭环。

### 4.2 后续版本

对话中明确提及的后续能力包括：

- 任务下发与执行状态机；
- 实际任务完成率；
- 机器人当前位置仅在实际到达后更新；
- 更丰富的时间范围封闭指令；
- 在机器人规模或求解要求提升时，将优先级规划替换或升级为 CBS/ECBS；
- 可选的 D* Lite 等其他路径规划算法；
- 不同机器人尺寸、速度或转向成本；
- 更完整的部署、接口、权限和可观测性体系。

### 4.3 不得擅自扩展

Agent 不得因为实现方便而：

- 恢复已删除的随机场景评估模块；
- 将 LLM 生成的路径直接视为有效路径；
- 将固定地图改为随机地图或自由拓扑图；
- 将部分成功改回“任一失败则整批失败”；
- 在没有执行反馈时计算实际任务完成率；
- 将规划成功自动视为机器人已经完成任务。

## 5. 产品原则

### 5.1 LLM 理解，算法求解

LLM/Agent 负责：

- 理解自然语言；
- 生成符合 Schema 的任务；
- 解析临时封闭约束；
- 根据冲突报告选择重规划策略；
- 输出可解释说明。

确定性模块负责：

- A* 搜索；
- 障碍物判断；
- 时空预留；
- 冲突检测；
- 路径验证；
- 指标统计。

不得让模型自由生成未经验证的路径。

### 5.2 不猜测缺失坐标

自然语言未明确提供的坐标只能来自：

1. 地图中的语义位置解析；
2. 机器人运行时状态；
3. 已明确的配置或调用参数。

否则必须返回缺失或未知位置错误，不得让模型编造坐标。

### 5.3 安全结果优先于任务数量

部分执行允许存在，但任何被标记为可执行的任务集合必须重新规划并重新验证。不得通过简单删除冲突任务后直接复用旧路径。

### 5.4 结果可解释、可审计

每次重规划应记录：

- 冲突类型；
- 受影响机器人；
- 选择重规划的机器人；
- 优先级变化；
- 新增约束；
- 重规划结果；
- 是否产生新冲突。

### 5.5 规划与执行分离

MVP 只负责路径规划。规划成功不等于任务已经执行或完成，运行时位置不得仅因规划成功而写回为终点。

## 7. 核心业务流程

### 7.1 加载地图与运行时状态

**触发条件：** 系统启动或处理新调度请求。

**前置条件：**

- 固定地图 JSON 存在且通过 Schema 校验；
- 机器人运行时状态可用，或调用方提供机器人当前位置；
- 用户指令中的临时封闭可被解析为地图通道或坐标集合。

**主要步骤：**

1. 加载固定地图；
2. 校验地图尺寸、坐标系统、静态障碍、语义位置、入口格和通道；
3. 加载机器人当前位置和已有临时封闭；
4. 合并用户指令中的临时封闭；
5. 生成本次请求的有效障碍与约束视图。

**失败处理：**

- 地图格式错误：拒绝规划并返回配置错误；
- 坐标越界或引用不存在：拒绝相关约束或任务；
- 机器人未知或禁用：相关任务规划失败。

### 7.2 自然语言任务解析

**触发条件：** 用户提交自然语言调度指令。

**主要步骤：**

1. 识别机器人任务；
2. 提取机器人 ID、起点、目标语义位置和优先级；
3. 解析临时封闭通道或坐标区域；
4. 通过地图位置注册表解析目标；
5. 对缺失起点使用机器人运行时位置；
6. 对多入口目标保留候选目标集合；
7. 执行 Schema 和业务校验。

**成功结果：**

- 结构化任务批次；
- 运行时约束列表；
- 解析警告；
- 可验证的错误列表。

**失败处理：**

- 未知机器人；
- 未知位置；
- 缺失起点或目标；
- 优先级非法；
- 坐标越界；
- 起点或所有候选终点被阻塞。

### 7.3 初始规划

**目的：** 为每个任务生成独立路径，并保留初次冲突统计能力。

**主要步骤：**

1. 每台机器人独立执行 A*；
2. 初始阶段考虑静态障碍和有效临时封闭；
3. 初始阶段不使用其他机器人的时空预留；
4. 输出每台机器人的时空路径；
5. 调用冲突检测器。

**失败处理：**

- 单机路径不存在；
- 起点或目标被封闭；
- 最大规划时域超限。

### 7.4 冲突检测

必须检测：

- 起点冲突；
- 终点冲突；
- 同一时间步占据同一格子的顶点冲突；
- 同一时间段交换相邻位置的交换冲突。

机器人到达终点后持续占用终点，因此较短路径在冲突检测中需要按终点位置补齐。

### 7.5 反思与最多 3 次重规划

**触发条件：**

- 初始路径存在冲突；
- 单机规划失败但可能通过目标选择、优先级调整或约束变化解决。

**已确认规则：**

- 重规划最多 3 次；
- 可以调整机器人优先级；
- 可以固定高优先级机器人路径；
- 可以通过预留表为低优先级机器人重新规划；
- 可以使用等待和绕行；
- 3 次后仍无法得到安全路径的任务不得标记成功。

**当前倾向的重试策略：**

1. 第一次：固定高优先级路径，对冲突中的低优先级机器人进行时空 A*；
2. 第二次：扩大冲突点时间窗口约束，增加等待或绕行；
3. 第三次：调整局部优先级，对受影响机器人重新规划。

该三级策略是建议方案，不得在未验证现有实现的情况下被视为唯一合法实现。任何替代实现仍必须满足最多 3 次、结构化诊断和最终验证要求。

### 7.6 部分执行

**触发条件：** 全部任务在最多 3 次重规划后仍无法共同形成安全方案。

**已确认规则：**

- 允许部分执行；
- 优先目标是最大化可完成任务数量；
- 同等完成数量下优先保留高优先级任务；
- 未执行机器人不会消失，必须持续占用其当前位置；
- 选定部分任务集合后必须重新规划并重新验证。

对于 3 至 5 台机器人，当前倾向是枚举或有限搜索任务子集。

### 7.7 输出结果与指标

批次状态应至少区分：

- `success`：全部任务规划成功；
- `partially_successful`：部分任务获得安全路径；
- `infeasible`：没有可执行的安全任务集合，或所有任务均失败。

MVP 必须输出路径规划成功率：

`路径规划成功率 = 最终获得无冲突路径的任务数 / 用户输入任务总数`

最终版本增加：

`实际任务完成率 = 实际到达目标并完成任务的任务数 / 用户输入任务总数`

## 8. 关键业务规则

1. 地图必须是固定的二维纯网格。
2. 坐标格式统一为 `[x, y]`。
3. 坐标原点、x 方向和 y 方向必须在地图 JSON 中明确。
4. 机器人只能上、下、左、右移动。
5. 机器人允许原地等待。
6. 所有机器人占据一个网格。
7. 所有机器人速度一致，每个时间步最多移动一格。
8. MVP 不考虑转向成本。
9. A* 内部正式路径必须包含时间维度 `(x, y, t)`。
10. 用户指令可以包含临时封闭通道或临时封闭坐标。
11. 未明确持续时间的临时封闭，在 MVP 中默认对本次规划全过程有效。
12. 用户指令中的临时封闭只能增加本次请求的不可通行约束，不得解除固定静态障碍。
13. 机器人到达终点后持续占用终点，直到本次调度结束或规划时域结束。
14. 同一个终点格不能分配给多台持续占用的机器人。
15. 语义设施本体格和机器人可停靠入口格应区分。
16. 对多入口目标，应保留候选目标集合，由调度器选择具体目标格。
17. 初始规划必须与重规划分开统计。
18. 初始独立规划不考虑其他机器人的时空预留。
19. 最终成功结果必须不存在顶点冲突和交换冲突。
20. 重规划最多执行 3 次。
21. 3 次后仍无法获得安全路径的任务必须标记失败或进入部分执行选择。
22. 部分执行时，未执行机器人持续占用当前位置。
23. 部分执行方案必须重新规划并重新验证。
24. 规划成功不得自动更新机器人真实当前位置。
25. 只有最终版本接收到实际执行完成反馈后，才能统计实际任务完成率。

## 9. MVP 范围与优先级

### 9.1 MVP 必须包含

#### 固定 JSON 地图加载

最小范围：

- 地图宽高；
- 坐标系统；
- 静态障碍；
- 语义位置；
- 目标入口格；
- 可选通道定义；
- 移动规则。

验收条件：

- 非法坐标、重复 ID、冲突别名和不可达入口可以被识别；
- 地图加载后转换为内部高效结构。

#### 自然语言结构化输出

最小范围：

- 机器人 ID；
- 起点；
- 目标位置或候选目标；
- 优先级；
- 临时封闭约束；
- 解析警告和错误。

验收条件：

- 不编造坐标；
- 缺失信息能从地图和运行时状态补全；
- 无法补全时返回结构化错误。

#### 时空 A*

最小范围：

- 状态 `(x, y, t)`；
- 四方向移动；
- 等待动作；
- 静态障碍；
- 临时封闭；
- 顶点和边预留；
- 最大规划时域。

验收条件：

- 路径连续；
- 不穿越障碍；
- 不违反临时封闭；
- 返回成功、代价、耗时和失败原因。

#### 冲突检测

最小范围：

- 起点冲突；
- 终点冲突；
- 顶点冲突；
- 交换冲突；
- 到达终点后持续占用。

验收条件：

- 冲突输出包含类型、机器人、时间和位置或边。

#### 反思与重规划

最小范围：

- 冲突原因分类；
- 优先级或预留约束调整；
- 低优先级机器人重规划；
- 最多 3 次；
- 重规划历史。

验收条件：

- 每次重规划有结构化决策和结果；
- 3 次后不会继续无限重试。

#### 部分执行

最小范围：

- 全量失败后搜索可执行任务子集；

## 10. 系统架构

### 10.1 已确认

总体采用"LangGraph 编排 + 确定性工具"的分层方式。调度流程由编译后的 `StateGraph` 驱动，每个处理阶段为独立节点函数。“Agent 编排 + 确定性工具”的分层方式。

```text
自然语言指令
    ↓
parse_instruction 节点 (TaskParserAgent)
    ↓
validate_and_resolve_goals 节点
    ↓
build_obstacles 节点
    ↓
initial_plan 节点 (独立 A*，无互斥预留)
    ↓
conflict_check 节点 (ConflictDetector)
    ↓  ←──┐
    ├─ 无冲突 → validate_final → compute_metrics → END
    ├─ 有冲突 → replan_decide → apply_replan ──┘ (最多循环 3 次)
    └─ 不可解 → partial_execution → validate_final → compute_metrics → END
```

核心模块：

- `app/orchestration/graph_builder.py`：`build_graph()` 构建 StateGraph，定义节点、条件路由和循环边
- `app/orchestration/graph_nodes.py`：10 个节点函数（纯函数，State → Partial Update）
- `app/orchestration/workflow.py`：`Workflow` 封装编译后 Graph，对外保持 PlanningState 接口
- `app/domain/graph_state.py`：`GraphState` TypedDict（22 字段，累积字段使用 operator.add reducer）
- `app/agents/task_parser_agent.py`：自然语言→结构化任务（LLM）
- `app/agents/replanning_agent.py`：冲突诊断与重规划决策
- `app/orchestration/replanning_policy.py`：执行重规划策略
- `app/tools/astar_planner.py`：确定性时空 A*
- `app/tools/conflict_detector.py`：顶点/交换/起点/终点冲突检测
- `app/tools/reservation_table.py`：时空顶点和边预留表
- `app/tools/path_validator.py`：最终路径有效性验证
- `app/services/map_loader.py`：固定 JSON 地图加载与校验
- `app/services/location_resolver.py`：语义位置与别名解析
- `app/services/robot_registry.py`：机器人运行时位置
- `app/services/metrics_collector.py`：规划指标（保留用于独立测试）
- `app/api/server.py`：FastAPI HTTP 服务器（调度/地图/运行时 CRUD Endpoint + React 前端服务）
- `app/api/schemas.py`：Pydantic API Schema（SchedulingRequest/Response）
- `frontend/`：React 18 + Vite Web 前端（地图编辑器 + NL 输入 + 路径可视化）

### 10.2 当前倾向

- 使用 LangGraph `StateGraph` 作为编排引擎；
- 通过编译后的 Graph 执行全流程，每个阶段为独立纯函数节点；
- 使用优先级规划法和预留表处理 3 至 5 台机器人；
- 对部分执行使用有限任务子集搜索；
- 对外提供 FastAPI REST API + React Web UI；
- 固定地图与运行时状态拆分为 `warehouse_map.json` 和 `warehouse_runtime.json`；

### 10.3 待确认

- 编程语言和具体 Agent 框架（当前：Python 3.9+，LangGraph 1.x）；
- 是否替换为更高层框架（如 LangChain 全栈）；
- 模型供应商和模型调用方式（当前：DeepSeek via OpenAI SDK）；
- 数据持久化方式；

## 14. 状态机与生命周期

### 14.1 规划状态

建议状态流：

```text
RECEIVED
→ PARSED
→ VALIDATED
→ INITIAL_PLANNED
→ CONFLICT_CHECKED
→ REPLANNING（可重复，最多 3 次）
→ SUCCEEDED / PARTIALLY_SUCCEEDED / INFEASIBLE
```

规则：

- 未通过解析或校验不得进入规划；
- 未通过最终路径验证不得进入成功状态；
- `REPLANNING` 次数不得超过 3；
- 部分成功必须基于重新规划和验证后的任务子集。

### 14.2 最终版本执行状态

后续版本计划：

```text
PENDING
→ PLANNED
→ DISPATCHED
→ EXECUTING
→ COMPLETED
```

异常状态：

```text
PLANNING_FAILED
EXECUTION_FAILED
CANCELLED
```

只有 `COMPLETED` 可以计入实际任务完成率。

## 17. 开发规范

以下为与当前项目直接相关的规则。

### 17.1 命名和类型

- 坐标统一使用 `x`、`y`，不得与 `row`、`column` 混用；

- 未知语义位置；
- 缺失起点；
- 缺失目标；
- 优先级非法；
- 坐标越界；
- 起点被阻塞；
- 目标被阻塞；
- 通道封闭；
- 无路径；
- 最大时域超限；
- 起点冲突；
- 终点冲突；
- 顶点冲突；
- 交换冲突。

### 17.4 日志和审计

建议默认规则：

- 记录请求 ID、规划阶段、耗时、冲突数量和重试次数；

- 固定地图不得硬编码在算法代码中；
- 最大规划时域应可配置；
- 未确认的默认值必须在配置或文档中显式标记；
- 用户指令中的临时封闭不应写回固定地图文件。

### 17.6 依赖和重构

- 新增依赖前先检查现有依赖；
- 不得仅因个人偏好替换 Agent 框架或算法库；

- 若实际目录与建议目录不同，优先最小改动。

MVP 不要求随机生成 100 组场景，也不要求场景级批量评估报告。可以保留针对固定地图的确定性回归用例，但不得将其包装为已确认的随机评估模块。

### 19.5 提交前

Agent 必须运行所有当前仓库适用的：

- 格式化；
- Lint；
- 类型检查；
- 单元测试；
- 相关集成测试。

无法运行时必须说明原因。

## 20. 安全、隐私与合规

当前项目未确认账号体系或合规标准，但仍必须遵守：

- 不得提交模型密钥、访问令牌、真实账号或凭据；
- 不得在日志中记录密钥；
- 地图和机器人状态属于运行数据，不得无说明发送给未批准的第三方；
- 自然语言输入属于不可信输入，必须做长度、格式、Schema 和引用校验；
- LLM 输出不得直接执行，必须经过结构化校验和确定性验证；
- 临时封闭指令不得用于解除固定静态障碍；
- 不得用真实生产机器人数据作为默认测试夹具；
- 不得执行未经说明的破坏性运行状态写回；
- 规划成功不得伪装成执行完成。

认证、授权、数据留存和审计合规待确认。

## 21. 性能、可靠性与可观测性

### 21.1 已知规模

- 机器人数量：3 至 5 台；
- 地图：固定纯网格；
- 重规划：最多 3 次。

### 21.2 性能指标

已确认需要记录：

- 总规划耗时；
- 路径规划成功率。

建议记录：

- 解析、初始规划和重规划耗时；
- 单任务平均规划耗时；
- A* 调用次数；
- 展开节点数；
- 冲突数量；
- 重试次数。

没有已确认的耗时上限、并发量或 SLA，不得自行写入数值目标。

### 21.3 可靠性规则

- A* 必须有最大规划时域，避免允许等待时无限搜索；
- 最大时域的具体默认值待确认；
- 所有成功结果必须再次经过路径验证；
- 重规划不得超过 3 次；
- 部分成功必须重新验证；
- 同一输入和同一地图在没有模型随机性的确定性模块中应尽量可复现。

### 21.4 可观测性

建议至少提供：

- 请求 ID；
- 阶段耗时；
- 冲突摘要；
- 重规划次数；
- 单任务状态；
- 失败代码；
- 最终指标。

监控、追踪、告警和日志平台待确认。

## 22. Agent 工作规则

### 22.1 开始任务前

Agent 必须：

1. 阅读根目录及相关子目录中的 `AGENTS.md`。
2. 阅读与任务相关的现有代码、测试和文档。
3. 确认任务是否属于当前 MVP 范围。
4. 识别受影响的领域模型、工具、Agent、编排、配置和测试。
5. 检查是否存在与当前任务冲突的既定决策。
6. 区分已确认需求、当前倾向、建议方案和自身假设。
7. 优先寻找现有实现，不得无必要重复建设。
8. 检查是否会恢复已删除的随机场景评估设计。
9. 检查是否会混淆规划成功与实际完成。

### 22.2 实现过程中

Agent 必须：

1. 优先进行满足需求的最小改动。
2. 遵守当前架构和模块边界。
3. 不得因为个人偏好替换技术栈。
4. 不得无任务依据进行大规模重构。
5. 不得让 LLM 替代 A*、冲突检测或最终验证。
6. 不得绕过类型检查、输入校验、安全校验或测试。
7. 不得通过删除测试使构建通过。
8. 新增依赖前必须说明现有依赖为何不足。
9. 修改业务逻辑时同步更新测试。
10. 修改接口、数据模型、配置或使用方式时同步更新文档。
11. 对无法确认的需求采用影响最小、可回退的实现，并明确记录假设。
12. 不得提交密钥、访问令牌、真实账号或真实生产数据。
13. 对运行状态写回等破坏性操作设置明确保护，不得默认执行。
14. 确保未执行机器人在部分执行方案中仍持续占用当前位置。
15. 确保到达终点的机器人持续占用终点。
16. 确保重规划次数不超过 3。
17. 确保成功路径通过最终验证。

### 22.3 完成任务后

Agent 必须：

1. 运行适用的格式化、Lint、类型检查和测试。
2. 汇总修改内容和原因。
3. 说明影响的模块和数据模型。
4. 说明是否改变接口、配置、路径语义或指标。
5. 报告已运行的测试及结果。
6. 报告无法运行的测试及原因。
7. 列出残留风险和待处理事项。
8. 检查实现是否超出 MVP 范围。
9. 检查是否引入与既定产品原则冲突的行为。
10. 检查文档是否仍与代码一致。

## 23. 禁止事项

Agent 不得：

- 擅自扩展产品范围；
- 擅自恢复随机场景生成和 100 组评估模块；
- 擅自修改固定纯网格、四方向和同构机器人规则；
- 擅自取消到达终点后的持续占用；
- 擅自取消部分执行；
- 擅自把规划成功率改成场景成功率；
- 擅自在 MVP 中计算实际任务完成率；
- 擅自用 LLM 生成路径并绕过确定性验证；
- 擅自修改最大 3 次重规划限制；
- 擅自改变坐标语义；
- 擅自将未执行机器人从地图中移除；
- 擅自替换 A* 或优先级规划核心方向；
- 擅自进行大规模重构；
- 擅自引入数据库、消息队列、缓存或外部服务；
- 擅自将建议项视为确定需求；
- 擅自编造运行命令、版本、端口、环境变量或部署平台；
- 擅自提交密钥或真实生产数据。

## 24. Definition of Done

一个项目任务只有在相关条件全部满足后才算完成：

- 实现符合已确认需求；
- 未超出 MVP 边界；
- 正常流程可用；
- 相关异常和边界场景得到处理；
- LLM 输出经过 Schema 校验；
- 成功路径经过地图、连续性、起终点和冲突验证；
- 重规划次数不超过 3；
- 部分执行正确处理未执行机器人占位；
- 到达终点后的持续占用得到处理；
- 路径规划成功率计算正确；
- 类型检查通过，若项目已配置；
- 适用单元测试和集成测试通过；
- 未通过删除测试规避失败；
- 接口、数据结构或配置变更已同步文档；
- 新增配置有明确说明；
- 未提交敏感信息；
- 无法验证的部分已明确说明；
- 残留风险和待确认事项已记录。

## 25. 已知风险和技术债务

### 25.1 优先级规划不完备

当前第一版采用优先级规划和预留表。该方法可能在存在可行解时，因为优先级顺序不佳而未找到解。

缓解方式：

- 最多 3 次优先级调整和局部重规划；
- 部分执行；
- 未来在需要更高完备性时评估 CBS/ECBS。

### 25.2 LLM 解析不确定性

自然语言可能存在歧义、别名冲突或缺失坐标。

缓解方式：

- 语义位置注册表；
- 别名索引；
- Schema 校验；
- 运行时位置补全；
- 不猜测未知坐标；
- 结构化警告和错误。

### 25.3 固定最大规划时域

允许等待会扩大时空搜索空间，必须设置最大时域。具体默认值尚未确认，过小会误判不可行，过大会增加耗时。

### 25.4 部分执行组合复杂度

当前规模为 3 至 5 台，有限子集搜索可接受；规模扩大后需要更高效的选择策略。

### 25.5 运行时状态与固定地图边界

固定地图、机器人位置和临时封闭的文件拆分尚为当前倾向，实际持久化方式待确认。

### 25.6 规划与真实执行存在差距

MVP 不处理机器人偏离路径、通信失败、执行中新增障碍等问题。实际任务完成率必须等待执行闭环。

### 25.7 技术栈和部署未定

缺少已确认的语言、框架、命令、模型服务和部署平台，会影响代码骨架、自动化和运维设计。

## 27. 历史决策与废弃方案

### 27.1 随机场景评估模块

曾计划在 100 组随机起终点、3 至 5 台机器人和随机临时封闭通道场景下统计场景成功率、初次冲突率、重规划触发率和平均耗时。

最终决定：

- 完全移除场景生成器和 100 组随机评估模块；
- 地图固定从 JSON 读取；
- 路径规划成功率基于用户输入任务数；
- 平均规划耗时等不依赖随机场景的指标可以保留。

### 27.2 整批失败与部分执行

曾讨论“任一机器人失败则整批失败”与“按机器人统计”两种方式。

最终决定：

- 允许部分执行；
- 仍保留批次状态；
- 未执行机器人持续占用当前位置；
- 部分执行方案必须重新规划和验证。

### 27.3 二维 A* 与时空 A*

普通二维 A* 无法可靠处理等待、交换冲突和时间范围封闭。

最终选择：

- 正式路径规划使用时空 A*，状态为 `(x, y, t)`；
- 允许等待动作。

### 27.4 LLM 直接规划与确定性工具

曾明确比较模型直接生成路径与确定性算法方案。

最终选择：

- LLM 只负责理解和决策；
- 路径搜索、冲突检测和验证由确定性工具完成。

### 27.5 规划成功率与实际任务完成率

最终决定：

- MVP 只保留路径规划成功率；
- 最终版本同时提供路径规划成功率和实际任务完成率；
- 未接入执行回传前不得计算实际任务完成率。
